# The resource graph

Internally, Lithos models your project as a directed acyclic graph of
*resources*. Every individual thing Roblox exposes — an experience, a
place, a place's configuration, a developer product, a thumbnail — is one
node. Edges encode "this resource needs that resource's outputs to exist
before it can be created."

You will rarely look at the graph directly, but understanding the model
explains every behaviour you'll see in the CLI output.

## A resource has four parts

| Part | Description |
| --- | --- |
| **Type** | What kind of Roblox thing it is (`place`, `developerProduct`, `thumbnail`, …). |
| **ID** | A stable identifier within your project, e.g. `place_start`. Lithos generates these from your config. |
| **Inputs** | The fields you set in `lithos.yml` plus the outputs of resources it depends on. |
| **Outputs** | What Roblox returned after creating it (asset IDs, version numbers, …). Outputs feed the inputs of dependent resources. |

A *change* is any difference between a resource's persisted inputs and its
freshly-computed inputs. Lithos uses this to decide between
**create**, **update**, **noop**, and **delete**.

Steps 1–4 are read-only (green). Nothing on Roblox changes until you
confirm at step 5 (amber). This is what makes destructive mistakes
recoverable: you always see them coming.

## Operation kinds

Each node in the diff is exactly one of these:

- `+` **create** — exists in config, not in state.
- `~` **update** — exists in both, inputs differ.
- `−` **delete** — exists in state, not in config.
- `=` **noop** — inputs match. No API call.
- `↷` **skip** — its dependencies failed; cannot proceed.

In CI logs you will see lines like:

```text
Succeeded with 0 create(s), 1 update(s), 0 delete(s), 5 noop(s), 0 skip(s)
```

That summary is the *only* number that matters when reviewing a deploy in
hindsight: a deploy with `0 delete(s)` and `0 skip(s)` was fully successful
and non-destructive.

## Why deletes are never silent

If you remove a resource from `lithos.yml`, the next deploy will plan a
delete. That is by design — the YAML is the source of truth, and the
absence of a resource means *delete it*.

What is **not** silent:

1. The plan preview marks the destructive operation with an explicit
   warning before any API call.
2. In an interactive terminal, Lithos prompts for confirmation. Cancel, and
   nothing happens.
3. In CI, deletes are still applied (otherwise you could never decommission
   anything from a pipeline), but they are clearly logged.

<Callout type="warning">
  Deletes from a Lithos perspective are also archives from a Roblox
  perspective. Roblox does not generally permit hard-deleting assets, so
  Lithos archives them and removes them from state. They will not come back
  on the next deploy unless you re-add them to your config.
</Callout>

## Drift, and what Lithos does about it

If someone — a teammate, a moderator, your past self — changes a Roblox
resource through the Creator Hub or Studio, the state file no longer
matches reality. Lithos calls that *drift*.

During the live reconciliation pass (step 2 above), Lithos asks Roblox to
confirm that each resource it thinks exists actually exists, classifying
each one:

- **verified** — Roblox confirmed it. Use the persisted state as-is.
- **missing** — Roblox said it does not exist. Drop it from state so the
  next plan re-creates it.
- **skipped** — no cheap, authoritative existence check (e.g. thumbnails).
  Trust the persisted state.
- **unknown** — the API call itself failed (network, permission). Trust
  the persisted state to fail safe.

The net effect: if a place gets deleted in the dashboard, the next deploy
re-creates it instead of producing a broken update against a non-existent
asset. If the API is flaky, Lithos does not destroy your state.
